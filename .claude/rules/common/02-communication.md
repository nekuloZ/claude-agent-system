# 02-communication.md - 沟通与输出规范

> **Rule Level:** L3 High Priority
> **Last Updated:** 2026-03-18
> **Applies to:** All Modes

---

## 🎯 Core Principle

**"工作讨论像聊天，文档说明要清晰"**

---

## 💬 工作讨论场景

### DO
- ✅ 使用自然对话（段落和句子）
- ✅ 简洁直接，先说结论
- ✅ 必要时提供简要解释

### DON'T
- ❌ 过度结构化（PPT风格、大标题、emoji堆砌）
- ❌ 长篇大论的解释
- ❌ 机械化回复

**Good Example:**
```
这个问题可能是路径不对，你检查一下配置文件里的路径
写的是相对还是绝对。如果是相对路径，当前工作目录可能不对。
```

**Bad Example:**
```
【问题分析】
1. 路径配置可能有误 📋
2. 需要检查配置文件
3. 确认相对/绝对路径
```

---

## 📄 文档输出场景

### 需要结构化的情况
- 代码示例
- API 文档
- 错误排查步骤
- 数据统计表格
- 配置文件

### 格式规范
- 使用 Markdown 标准格式
- 表格用于对比/清单
- 代码块带语言标识
- 章节层级清晰（# ## ###）

---

## 📤 第三方消息场景

**当生成发给第三方的消息时（客户/同事/外包）：**

1. **自然段落形式** - 避免结构化文本（分点列表）
2. **说明上下文** - 让接收方了解情况背景
3. **说明诉求** - 明确表达你要什么
4. **语气自然** - 像平时说话一样
5. **避免过度格式化** - 不要用太多 emoji、大标题、粗体

**Example:**
```
你好，关于项目进度的事想跟你确认一下。之前我们约定的时间是月底，
但目前看可能要延期。另外费用方面老板问预算会不会超。质量方面我们
还是按之前的标准。你什么时候方便，我们电话聊一下？
```

---

## 🌏 语言与本地化

### 默认语言
- **沟通语言：** 简体中文
- **代码注释：** 中文说明 + 英文关键字
- **日志输出：** 英文（方便搜索排错）

### 中英混排规范
- 优先：`转化率 conversion_rate`
- 保留英文：函数名、变量名、API、错误信息、HTTP状态码

## 📐 输出结构模板

### 简短回答
```
[结论/答案]

[1-2句必要解释]
```

### 技术回复
```
[结论]

```代码/配置```

[简要说明关键部分]
```

### 复杂回复
```
## [标题]

[概述]

### [子主题]
[内容]

### [子主题]
[内容]
```

---

## ⚡ 快速回复规则

| 场景 | 回复长度 | 格式 |
|------|----------|------|
| 简单确认 | 1-2行 | 纯文本 |
| 状态报告 | 3-5行 | 简洁列表 |
| 技术方案 | 段落+代码 | 结构化 |
| 问题分析 | 分层说明 | 章节化 |
